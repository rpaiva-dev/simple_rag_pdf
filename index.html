Get started How RAG works

What this project does

Quarterly IR reports are long PDFs packed with tables. Finding one specific number means flipping through dozens of pages — this project turns that into a single question.

📎

Attach a report

Upload the PDF of an earnings release or a REIT management report, or paste the link of the IR page — the system finds and downloads the report by itself.

💬

Ask in plain language

"What was the net income this quarter?", "How much did the fund distribute per share?", "What is the current vacancy rate?" — no need to know where the answer lives.

📌

Verify the source

Every answer cites the document, page and section it came from. If the information is not in the report, the system says so — it never invents a financial figure.

What is RAG? (in plain terms)

RAG stands for Retrieval-Augmented Generation — a technique that makes an AI answer based on your documents, not on what it memorized during training.

The open-book exam analogy: a regular AI chat is like a student taking a closed-book exam — when it doesn't remember, it guesses (confidently). RAG turns it into an open-book exam: before answering, the system searches your documents for the relevant passages, hands them to the AI and orders: "answer using only this, and say where it came from."

🗂️

1 · Indexing

The document is split into small pieces ("chunks"), and each piece is converted into a list of numbers (an embedding) that captures its meaning. Similar meanings become similar numbers.

🔍

2 · Retrieval

Your question is also converted into numbers, and the system finds the pieces of the document whose meaning is closest to the question.

✍️

3 · Generation

The retrieved pieces are handed to the language model along with strict rules: use only this context, cite the source, and never estimate numbers that are not literally written there.

Architecture: from PDF to answer

Every step of the pipeline is a small, readable Python module — built by hand precisely to expose what RAG frameworks hide under the hood.

1

Extraction — src/extract.py

Reads the PDF with pdfplumber, separating running text from tables. A financial table mixed into text scrambles labels and values — keeping them apart is what prevents confusing "net income" with "vacancy". Links to IR pages are resolved and downloaded automatically.

▼
2

Chunking — src/chunking.py

Splits the text into ~400-token pieces, cutting by section first ("Financial Results", "Dividend Distribution"...), then by paragraph, with overlap between neighbors. Tables are never split. Each chunk carries its metadata — document, page, section — which is where source citation comes from.

▼
3

Embeddings — src/embeddings.py

Converts each chunk into a 384-number vector using the all-MiniLM-L6-v2 model (runs locally, free). Vectors are normalized to simplify similarity comparison in the next step.

▼
4

Vector store — src/vector_store.py

Cosine-similarity search in pure NumPy — no FAISS, no Chroma. At the scale of a few reports, one matrix multiplication is instant, and the whole store fits on one screen: exactly what "real" vector databases do under the hood.

▼
5

Retrieval — src/retrieval.py

Turns the question into a vector and fetches the 5 most similar chunks. If none passes the minimum similarity threshold, the system concludes the answer is not in the document — an honest "I don't know" beats an invented number.

▼
6

Prompt assembly — src/prompt_builder.py

Joins the retrieved passages (with page and section visible) to the question, under explicit rules: answer only from the context, never round or estimate a number, and cite the source of every fact.

▼
7

LLM — src/llm_client.py

Sends the prompt to the OpenAI API (gpt-4o-mini) with temperature 0 — deterministic answers, no "creativity" around numbers. The hard work (finding the right information) was already done by the previous steps.

▼
8

Interface — app.py (Streamlit)

Chat with separate conversations (create, rename, delete), PDF attachment right in the chat, PT/EN language toggle and a "Sources used" panel on every answer showing document, page, section and similarity score.

How to run it

Requirements: Python 3.10+ and an OpenAI API key.

1. Clone the repository and create a virtual environment

git clone https://github.com/rpaiva-dev/investor_relations_analysis.git
cd investor_relations_analysis
python -m venv .venv

# Windows:
.venv\Scripts\activate
# Linux/Mac:
source .venv/bin/activate

2. Install the dependencies

pip install -r requirements.txt

The first install takes a few minutes — the embedding model relies on PyTorch, which is heavy. It only happens once.

3. Configure the API key

# copy the example file and edit it with your key
copy .env.example .env      # Windows
cp .env.example .env        # Linux/Mac

Inside .env: OPENAI_API_KEY=your-key-here

⚠️ Never share your key or push it to GitHub — the .env file is already protected by the project's .gitignore.

4. Start the app

streamlit run app.py

The browser opens automatically at http://localhost:8501.

How to use it

  1. Create a chat (or use the default one). Each chat holds its own documents and history — rename or delete it from the "⋯" menu.
  2. Attach the report PDF (.pdf only) in the uploader, or paste the IR page link in the chat. The question field only unlocks after a document is processed, and the chat is renamed automatically after the file.
  3. Ask in natural language — e.g. "What was the net income this quarter?", "How much did the fund distribute per share this month?", "What is the current vacancy rate?"
  4. Check the source: under each answer, the "📌 Sources used" panel shows the document, page and section the information was extracted from — you can open the PDF and verify it.
  5. Compare quarters: attach more than one PDF to the same chat and ask comparative questions ("did the distribution grow versus last quarter?").
💡 If you ask something that is not in the document (today's stock price, recent news), the system will answer that the information is not in the provided documents — that is the correct behavior, not a bug.
